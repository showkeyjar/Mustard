# Mustard 项目重设方案（回退到 Codex 人机协作）

## 1. 背景判断

当前项目的真实目标，不是把 `Claw Team` 做得更像一个组织，而是做出一个：

- 知识表示密度高
- 逻辑能力强
- 能在真实任务中稳定工作的智能体原型

从仓库现状看，`Claw Team` 已经演化成一套较重的“流程中台”，而不是加速核心能力进化的工具。它能持续生成日报、提案、角色产物与状态裁决，但这些产物和“智能体能力是否真的变强”之间的映射关系仍然偏弱。

因此，本轮建议不是继续优化团队编排，而是明确做一次研发模式重置：

- 从“多角色自动团队驱动”切回“Codex + Human 的高带宽闭环”
- 把 `Claw Team` 从主执行路径降级为辅助分析工具
- 把项目北极星重新绑定到能力验证，而不是流程完整度

## 2. 当前项目现状

### 2.1 可以肯定的正向积累

项目并不是空的，反而已经有一套相当完整的智能体脚手架：

- 核心能力层：
  - `carm/core.py`
  - `carm/runner.py`
  - `carm/policy.py`
  - `carm/memory.py`
  - `carm/experience.py`
- 训练与评测层：
  - `scripts/build_pretrain_dataset.py`
  - `scripts/pretrain_carm.py`
  - `scripts/auto_train.py`
  - `scripts/evaluate_pretraining.py`
  - `scripts/evaluate_real_prompts.py`
- 桌面桥梁层：
  - `carm/desktop.py`
  - `carm/bridge.py`
  - `scripts/desktop_agent_control.py`
  - `scripts/desktop_bridge_chat.py`

这说明仓库已经具备：

- 结构化状态表示
- 基础推理动作循环
- 经验回放与轻量适应
- 训练数据构建
- 标准评测与真实 prompt 回归
- 桌面桥梁原型

也就是说，项目最大问题不是“还没开始做能力”，而是“能力研发主线被流程主线盖住了”。

### 2.2 当前最明显的结构性问题

#### 问题 A：事实源分裂

当前至少存在三套彼此不同步的“项目状态”：

- 训练总报告：`data/train_runs/auto_train_latest.json`
- 最新真实回归：`data/eval/real_prompt_eval_latest.json`
- 团队角色产物：`backlog/role_outputs/*.md`

例如：

- `auto_train_latest.json` 仍停在 `20260324T094015Z`
- 其中 `real_prompt_eval.summary.pretrained_match_rate = 0.8333`
- 但 `data/eval/real_prompt_eval_latest.json` 已经是 `prompt_count = 20`、`pretrained_match_rate = 1.0`
- `backlog/role_outputs/benchmark_owner_latest.md` 也声称 `real_prompt_count: 20`

这会带来一个直接问题：团队可以持续宣布“方向正确”，但研发人员并不能一眼判断“当前哪个指标才是最新可信状态”。

#### 问题 B：流程复杂度已经高于能力复杂度

`scripts/team_conductor.py` 体量已经约 200 KB，且承担过多职责：

- 信号采集
- 提案生成
- 研究质量评估
- 递归状态控制
- 角色产物写入
- 交付决策
- GitHub 交付编排

这意味着：

- 流程系统本身已经成为主要维护对象
- 每多一个“研究恢复算子/角色规则/交付规则”，都会继续放大系统复杂度
- 团队产物越来越多，但能力增量越来越难追踪

#### 问题 C：成功标准偏流程、偏局部，不够贴近最终目标

当前团队非常强调：

- daily digest
- proposal completeness
- role output coverage
- top gap / failure pattern / research recovery

这些都不是没价值，但它们更像“研发运营指标”，不是“是否做出了接近 GPT-5.4 级逻辑体”的核心指标。

项目真正需要回答的问题应该是：

1. 它在哪些真实任务上比当前 baseline 更强？
2. 它在哪类复杂逻辑任务上仍然明显失效？
3. 失效来自：
   - 表示不足
   - 推理控制不足
   - 工具选择不足
   - 数据不足
   - 训练方式不足

而不是先问“本轮各角色是否完整出席”。

#### 问题 D：自动团队适合做维护，不适合做前期高不确定研发

当目标是“把一个原型系统持续维护并做低风险优化”时，多角色团队很合适。
但当前目标其实是一个高不确定、强研究型问题：

- 如何做更高密度的知识表示
- 如何做比当前轻量推理核更强的逻辑控制
- 如何把离线训练、在线纠偏和真实任务表现连成闭环

这类问题最需要的是：

- 快速提出技术假设
- 快速做小实验
- 快速看中间表征与失败样本
- 快速砍掉错误方向

这更适合高带宽的人机结对，而不是多角色文档流水线。

## 3. 我对当前项目阶段的判断

Mustard 现在不是“产品化打磨期”，而是“研究原型收敛期”。

在这个阶段，最合理的项目定义应该是：

> 在一个受控任务域内，做出一个明显优于当前 baseline 的高密度知识表示 + 可解释推理控制原型，并建立可重复验证的训练/评测闭环。

注意这里有三个关键词：

- `受控任务域`
- `明显优于 baseline`
- `可重复验证`

如果目标直接写成“逻辑能力媲美 GPT-5.4”，那对项目管理没有帮助，因为：

- 目标无法在短周期内证伪
- 容易把仓库推向过宽的技术面
- 会使每个子系统都想做“全能型增强”

更合理的方式是把大目标拆成可落地的阶段目标。

## 4. 建议的新目标设定

### 4.1 北极星目标

新的北极星目标建议改成：

> 在 `信息冲突判断 + 多源整合 + 工具选择` 三类核心逻辑任务上，做出一个比当前 Mustard baseline 明显更稳的 CARM 原型。

原因：

- 这三类任务已经在现有评测中出现
- 它们能较好代表“知识表示 + 逻辑控制”的核心难点
- 比追求“全场景接近 GPT-5.4”更可管理

### 4.2 三个阶段目标

#### Phase 1：做强评测与事实源

目标不是继续堆提案，而是建立一个唯一可信的研发看板：

- 一个主报告文件，能回答“当前最强版本是谁”
- 一个固定的 hard eval 集，覆盖高信息量逻辑任务
- 一个失败案例池，按根因聚类

交付标准：

- 任意时刻只看一个文件，就能知道当前最佳实验结果
- 新改动必须带前后对比
- 失败样本能映射到具体模块

#### Phase 2：做强推理内核，不扩展新外设

这个阶段建议把精力集中在：

- 推理状态表示
- 中间结构压缩与保真
- 工具选择判定
- 终止判断与验证动作

不建议继续扩：

- 更重的 Team 角色系统
- 更多 GitHub 自动交付流程
- 更复杂的桌面常驻行为

也不建议优先做：

- 默认运行时上线策略
- 激进主动追问
- 自动学习策略扩权

#### Phase 3：只在核心原型稳定后再回接桌面桥梁

桌面桥梁不是没有价值，而是它现在会把问题复杂度抬得过高：

- 观测噪声高
- 反馈成本高
- 行为风险高
- 很难判断失败来自 bridge 还是 core reasoning

所以建议：

- 桌面桥梁保留，但降为次级实验轨
- 默认主线只围绕离线任务 + 真实 prompt 回归推进
- 等核心逻辑原型稳定后，再回接桌面场景验证

## 5. 建议的新研发模式

### 5.1 模式切换

从：

- `Claw Team` 主导

切到：

- `Human + Codex` 主导
- `Claw Team` 辅助

具体定义如下：

#### Human 负责

- 最终目标裁剪
- 技术路线选择
- 高风险决策审批
- 判断某个实验是否值得继续

#### Codex 负责

- 阅读代码、找证据
- 实现最小改动
- 增补测试
- 跑训练/评测
- 直接总结结果与下一步建议

#### Claw Team 只保留两个用途

1. 周期性生成观察摘要
2. 在你明确要求时，辅助生成候选 failure pattern / proposal

也就是说，`Claw Team` 不再是“默认开发驾驶员”，而是“可调用的分析插件”。

### 5.2 新的默认工作流

建议默认采用下面这条 6 步路径：

1. 人类给出一个明确的能力问题
2. Codex 在代码中定位相关模块与现有评测
3. Codex 提出 1 个最小可验证改动
4. Codex 实现并跑测试/评测
5. 人类看结果，决定继续、回退或换方向
6. Codex 记录结论到单一事实源

这个流程的核心是：

- 一次只推进一个明确能力问题
- 不要求全角色同时“出席”
- 不为了流程完整而制造文档

## 6. 建议的新项目结构约束

### 6.1 重新定义主事实源

建议新增或重定义一个主状态文件，例如：

- `artifacts/current_best.json`

它应当成为唯一可信的当前状态入口，至少包含：

- best_run_id
- best_variant_name
- code_commit
- dataset_version
- eval_set_version
- hard_eval_score
- real_prompt_score
- key_failures
- changed_modules
- decision

之后：

- `auto_train_latest.json` 是训练流水线产物
- `real_prompt_eval_latest.json` 是局部评测产物
- `current_best.json` 才是项目事实源

### 6.2 精简团队层

建议把“长期常驻角色”缩成 3 个逻辑层，而不是完整团队：

- `Builder/Research`
  - 负责实验设计与代码落地
- `Evaluator`
  - 负责统一评测与结果记录
- `Guardian/Human Gate`
  - 负责高风险边界

以下角色不再作为默认执行单元：

- Observer
- Conductor
- Arbiter
- Trainer
- Benchmark Owner
- Failure Miner
- Researcher

它们可以保留为“文档视角”或“按需模板”，但不应继续作为每轮都要完整运转的流水线节点。

### 6.3 精简指标

默认只盯 4 个主指标：

1. `hard_logic_pass_rate`
2. `real_prompt_pass_rate`
3. `critical_failure_count`
4. `avg_steps_or_cost`

辅助指标才保留：

- prompt_count
- bridge_feedback
- frontier_observation_count
- role output completeness

也就是说，辅助指标不能再反客为主。

## 7. 技术路线上的更合理设定

### 7.1 不要把“媲美 GPT-5.4”理解成全面对齐

更合理的技术设定是：

- 不追求开放域全能
- 追求在特定逻辑能力束上形成明显优势
- 先做“结构优于 baseline”，再谈“总体逼近 frontier model”

建议把“高知识表示密度”具体化为三个可检验方向：

1. 更强的中间结构表示
   - 让 `PLAN/HYP/DRAFT` 不只是槽位，而是可压缩、可验证、可转训练信号的中间图式
2. 更强的冲突与证据管理
   - 明确记录 claim / source / contradiction / unresolved
3. 更强的终止与验证机制
   - 在“继续搜 / 应该停 / 需要验证”之间做更稳定判断

### 7.2 近期研发主线建议

未来 2-4 周，建议只围绕下面三条主线：

#### 主线 1：Conflict-aware reasoning

把冲突检测从“工具选对了”升级到“证据冲突被显式表示和处理了”。

#### 主线 2：Result integration with verification

把多源结果整合从“能总结”升级到“能标注哪部分确定、哪部分待确认”。

#### 主线 3：Termination and tool boundary

把“何时停、何时继续查、何时换工具”做成稳定规则或可学习控制。

这些主线都比扩展团队流程更接近项目真正价值。

## 8. 推荐的立即执行动作

### P0：本周就该做

1. 确定 `Claw Team` 降级为辅助工具，不再作为默认主流程
2. 设立单一事实源文件
3. 冻结一个真正高信息量的 hard eval 集
4. 从当前评测里挑出 10-20 个最有诊断价值的失败/险过样本
5. 只选一个核心能力问题作为下一轮改进目标

### P1：接下来 1-2 周

1. 重构评测汇总逻辑，让 `train/eval/current_best` 三者关系清晰
2. 把 `team_conductor` 从“大一统脚本”拆回更小职责单元
3. 围绕 conflict / integration / termination 做一次最小核心改造
4. 每次改动都要求给出失败案例前后对比

### P2：2-4 周后再考虑

1. 是否恢复部分自动团队角色
2. 是否扩大桌面桥梁闭环
3. 是否引入更重的预训练或状态空间模块

## 9. 最终建议

Mustard 当前最需要的，不是更多角色，也不是更多自动提案，而是一次明确的研发重心回正：

- 从“团队自己维护自己”回到“核心智能体能力进化”
- 从“流程完成度”回到“能力增量与失败根因”
- 从“泛化大愿景”回到“受控能力束突破”

如果只用一句话概括这次重设：

> 先把 Mustard 做成一个在少数关键逻辑任务上真正有锋芒的原型，再考虑把它包装成可持续自驱的团队系统。
